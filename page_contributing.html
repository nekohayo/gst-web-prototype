Welcome, developers!

If you are not already familiar with how the GStreamer library works, take a look at our wonderful Documentation page. We suggest to read the manual and the "introductory slides" to get a good general overview.

We would be happy to help you started. You are welcome to:

  • Contribute to the core library.
  • Write plugins for GStreamer.
  • Write applications using GStreamer.
  • Contribute to the documentation.
  • Build test cases.

The developers tend to live in the "#gstreamer" IRC channel on Freenode. More details and options for getting in touch can be seen on the contact page.

Of course, testers are welcome too! See the bugs page for details.

Anonymous Git access

The repository is split in several modules.

Those modules can be checked out through Git with the following instruction set. Replace "★" by the name of the module you wish to checkout. To determine the names of the modules created by this project, you may examine their Git repository via the provided web-based Git repository viewer.

git clone git://anongit.freedesktop.org/gstreamer/★

Frequent contributor Git access

Core project developers, needing to review/merge/push changes on a frequent basis, can access the Git tree via this method:

git clone ssh://user@git.freedesktop.org/git/gstreamer/★

Typically, you should only apply for a developer account if you have been encouraged by a GStreamer developer to do so, or have been contributing to GStreamer and been active in the GStreamer community for some time.

Indeed, you do not need a developer account in order to work on GStreamer. Since Git is a decentralized system, anonymous git access is usually sufficient as it's easy to maintain personal git branches elsewhere with git (e.g. on a personal server, or gitorious or github). Patches can and should then be submitted via bugzilla, as per the Submitting Patches page.

You can find information on how to get a developer account here. Developers will be added to the project for write access as necessary. For all other changes, please submit patches (in git format-patch format) to bugzilla. File a new bug and then attach the patch to the bug.

Developer Guidelines

Best practices for making patches and/or committing to GStreamer git can be found on the Developer Guidelines page.

Buildbot and automated media integration tests

Commits to Git are verified by our build and testing system. This does automated checkouts, runs the build and the various test suites, including media file tests. If a commit breaks a build, it is announced in the gstreamer developer channel and easily visible on the GStreamer Jenkins instance's web interface. This allows us to quickly catch any regressions.
